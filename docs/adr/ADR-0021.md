# ADR-0021 — Uniform connector abstraction behind an Integration Hub

- **Status:** Accepted
- **Phase:** 9

## Context
ForgeAI needs to work across many external systems (GitHub, Jira, Slack, Notion,
Confluence, Email, Calendar, Figma). Wiring each into agents directly would
couple agents to eight SDKs, make systems non-swappable, and scatter
permissions/secrets/approval logic.

## Decision
Every external system implements one `Connector` interface (search/get/create/
update over a normalized `ExternalObject`). All access goes through a single
`IntegrationHub` that enforces per-agent permissions, gates sensitive writes
behind approval rules, runs cross-system search, and feeds a `KnowledgeGraph`.
Secrets are encrypted at rest.

Offline `Fake*` connectors back the whole suite; production connectors (httpx
per vendor) implement the same interface and drop in unchanged.

## Consequences
- ✅ Agents treat all systems alike; any connector is replaceable (local search →
  vendor API, etc.).
- ✅ Permissions, approvals, and secret handling live in one place.
- ✅ Cross-system knowledge (graph + "why?" retrieval) becomes possible.
- ✅ The whole ecosystem is testable offline (extends the offline-first ADR line).
- ➖ The fakes approximate vendor APIs; each production connector still needs
  live validation (auth, pagination, rate limits) like GitHub did in Phase 8.1.
- ➖ Eight integrations is broad; depth per connector grows as real use demands.
