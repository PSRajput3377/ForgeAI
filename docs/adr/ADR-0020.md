# ADR-0020 — Author commits via a local clone + git, not the REST API

- **Status:** Accepted
- **Phase:** 8.1

## Context
GitHub's REST "contents" API can create commits, but real engineers work on a
local clone: clone → branch → edit → commit → push. Authoring commits over REST
diverges from that and doesn't compose with the rest of ForgeAI.

## Decision
ForgeAI authors all commits through a `LocalRepository` (the git CLI) operating
on a local clone. The REST provider (`RestGitHubProvider`) handles only refs,
PRs, reviews, and check runs — `create_commit` there is intentionally
`NotImplementedError`.

## Consequences
- ✅ Matches how engineers actually work; the working copy is a real git repo.
- ✅ Composes naturally with the Docker sandbox (run/test the clone), Reflection
  (fix in place), Review, and CI (push triggers checks).
- ✅ File writes go through the sandboxed FilesystemTool (path-escape safe).
- ✅ Testable offline with a bare local "remote" — no network, no token.
- ➖ Requires managing clone working directories (disk, cleanup) — handled per
  task/workspace.

## Related
Hardening done alongside this (Phase 8.1): rate-limit backoff + pagination for
the REST provider, and webhook ingestion (event-driven instead of polling).
Live validation runs against a **sandbox repo** via `scripts/verify-github.sh`.
