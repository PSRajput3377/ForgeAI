# ADR-0016 — Sandbox abstraction with offline backends for execution

- **Status:** Accepted
- **Phase:** 5

## Context
Autonomous execution needs real isolation (Docker), but the Docker daemon isn't
available in CI or on a fresh dev machine. The self-correcting loop (run →
classify → fix → retry) must still be fully testable, deterministically.

## Decision
Define a `Sandbox` interface with three implementations:

- `DockerSandbox` — production isolation (disposable container, `--network none`,
  CPU/memory/timeout limits).
- `LocalSandbox` — degraded host fallback when Docker is down (timeout +
  blocked-command rules still enforced; NOT isolated).
- `FakeSandbox` — scripted results for tests, so a run can simulate "fail twice,
  then pass" without executing anything.

The `ExecutionEngine` depends only on the interface and on an injected *fixer*,
so the entire reflection/retry loop is exercised offline.

## Consequences
- ✅ The whole execution loop runs in CI with no Docker (extends ADR-0011/0015).
- ✅ Real isolation in production; graceful degradation when Docker is absent.
- ✅ Security (blocked commands) enforced centrally, in every backend.
- ➖ Three sandbox implementations to maintain.
- ➖ `LocalSandbox` is not isolated — it is a fallback, and AI-generated code
  should prefer `DockerSandbox` (documented in security.md).
