# ADR-0019 — GitHub provider abstraction with an offline fake

- **Status:** Accepted
- **Phase:** 8

## Context
GitHub integration (branches, commits, PRs, reviews, CI) needs the real GitHub
API in production, but the autonomous workflow — especially the "CI fails → fix
→ re-run → merge" loop — must be testable with no token and no network.

## Decision
Define a `GitHubProvider` interface with two implementations:

- `RestGitHubProvider` — GitHub REST v3 via httpx + a Personal Access Token.
- `FakeGitHubProvider` — in-memory simulation of the full lifecycle, with the
  ability to **script CI outcomes** for deterministic tests.

`GitHubManager` orchestrates over the interface and takes an injected reviewer
and CI fixer (same decoupling as the Phase 5 ExecutionEngine). CI failure
analysis reuses the Phase 5 error classifier.

## Consequences
- ✅ The entire GitHub workflow runs offline (extends the offline-first line of
  ADRs 0011/0015/0016/0017/0018).
- ✅ Provider-independent: a GitLab/Bitbucket provider could be added later.
- ✅ CI self-correction is reused, not reinvented.
- ➖ The fake approximates GitHub; real-API integration (auth scopes, pagination,
  rate limits, commit authoring via local clone) still needs live verification.
- ➖ `RestGitHubProvider.create_commit` is intentionally not implemented via REST
  — real commits are authored with git against a local clone (the GitTool).
