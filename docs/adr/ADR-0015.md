# ADR-0015 — Memory & RAG: pluggable backends, offline by default

- **Status:** Accepted
- **Phase:** 4

## Context
The memory + RAG engine depends on Ollama (embeddings), Qdrant (vectors), and
Redis (cache) — none of which run in CI or on a fresh dev machine. We still need
the full pipeline (index → embed → store → retrieve → context) to be testable.

## Decision
Every component is defined behind an interface with two implementations:

| Concern    | Offline (default/tests) | Production            |
|------------|-------------------------|-----------------------|
| Embeddings | `HashingEmbedder`       | `OllamaEmbedder`      |
| Vectors    | `InMemoryVectorStore`   | `QdrantVectorStore`   |
| Cache      | `InMemoryCache`/`Null`  | `RedisCache`          |
| Memory     | `InMemoryStore`         | PostgreSQL (DB phase) |

Production backends import their clients **lazily** so the packages work without
those libraries installed. A factory in `apps/api/app/memory_runtime.py` selects
the real backends from settings.

The `HashingEmbedder` is a hashed bag-of-words producing meaningful cosine
similarity, so retrieval-quality tests are real (the right file is found by
meaning) without a model.

## Consequences
- ✅ The entire engine runs in CI with no external services (extends ADR-0011).
- ✅ Swapping/adding a backend is a one-class change; agents are unaffected.
- ➖ Two implementations per component to maintain.
- ➖ Offline embeddings are lexical, not semantic-deep — fine for plumbing/
  ranking tests; true semantic quality is measured by the evaluation harness.
