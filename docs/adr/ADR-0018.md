# ADR-0018 — Async SQLAlchemy with SQLite-in-memory for offline tests

- **Status:** Accepted
- **Phase:** 7

## Context
Phase 7 introduces the first real persistence (users, orgs, workspaces,
memberships, …). Production uses PostgreSQL, which isn't running in CI or on a
fresh machine — but the auth + multi-tenancy logic must be fully testable.

## Decision
Use **async SQLAlchemy 2.0** models that run unchanged on two backends, selected
by `DATABASE_URL`:

- **PostgreSQL** via `asyncpg` (production).
- **SQLite in-memory** via `aiosqlite` (tests).

Tests override the FastAPI `get_session` dependency with a SQLite session
factory. Tables are created from `Base.metadata` (Alembic migrations manage prod
schema). `python-jose` issues JWTs; `argon2-cffi` hashes passwords; the token
denylist is in-memory now (Redis later).

## Consequences
- ✅ The entire auth/tenancy suite runs offline (extends ADR-0011/0015/0016/0017).
- ✅ One model set, two backends; no test-only schema drift.
- ➖ SQLite ≠ Postgres in edge cases: it returns **naive** datetimes (we coerce
  to UTC-aware) and is laxer on types. Integration against real Postgres is still
  worthwhile before release.
- ➖ `create_all` on startup is a dev convenience; production must use Alembic.
