# ADR-0022 — Queue-based execution + reliability primitives behind interfaces

- **Status:** Accepted
- **Phase:** 10

## Context
Agent runs are long-running and depend on flaky external systems. A
request-blocking model doesn't scale to thousands of runs, and the platform
needs production reliability (retries, circuit breaking, rate limits, health,
metrics) — all testable without standing up Redis/Celery/Prometheus.

## Decision
Introduce a `packages/runtime/` layer of production primitives, each behind an
interface with an offline-testable implementation (injected clocks/sleeps):

- **Queue + worker pool** (`JobQueue`) — enqueue → workers → retry → DLQ. Mirrors
  Celery/RQ; in-memory now, Celery/Redis in production.
- **Reliability** — `RetryPolicy`/`run_with_retry`, `CircuitBreaker`,
  `DeadLetterQueue`.
- **Rate limiting** — token bucket per key (Redis-backed in prod).
- **Health + metrics** — `HealthRegistry` (readiness) + `MetricsRegistry`
  (Prometheus text), exposed at `/health/ready` and `/metrics`.
- **Feature flags** + **NetworkPolicy** (egress allow-list).
- **Parallelism** — split → run concurrently (bounded) → merge.

## Consequences
- ✅ Non-blocking, scalable execution model; reliability is first-class.
- ✅ Entire production layer is unit-tested offline (extends the offline-first
  ADR line through 0011/0015/0016/0017/0018/0019/0021).
- ✅ Operability: standard health/readiness/Prometheus surfaces for K8s + Grafana.
- ➖ The real Celery/Redis/Prometheus/Grafana/K8s wiring is deployment-time and
  not exercised here; interfaces are in place so it drops in.
- ➖ In-process queue ≠ distributed queue semantics (at-least-once, visibility
  timeouts); the Redis/Celery backend provides those.
