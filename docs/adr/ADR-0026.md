# ADR-0026 — A Project owns a workspace directory on disk

- **Status:** Accepted
- **Phase:** 13.1 (Projects & First-Run Experience)

## Context
Through Phase 12 the engine had no first-class notion of a *project*. `/agents/run`
took free-form `project_id` / `project_path` strings that linked to nothing, so
"Add JWT authentication" had no answer to "to which project?". The tool layer
already scopes to a workspace `root` (`build_default_registry(root)`), and the
DB already has `Organization → Workspace → Project` tenancy — but nothing tied a
`Project` row to a real directory the agents could read and write.

## Decision
A **`Project` owns a workspace directory on the API host**:
`<WORKSPACES_ROOT>/<project_id>`, created when the project is created and removed
when it is deleted. The `Project` model gains `path`, a nullable `repo` (so a
future git-backed mode slots in behind the same model with no agent/graph
change), and `starter` (for bootstrap, 13.3). A `ProjectService` provisions and
removes the directory; `/projects` CRUD endpoints enforce the existing workspace
RBAC. The path is **derived from the id**, never taken from the client, so a
project can only ever own its own folder under the configured root.

## Consequences
- ✅ Resolves the product's biggest gap: requests now run *against a real
  project*. Bootstrap (13.3) and the first-run flow (13.4–13.5) build on this.
- ✅ Path confinement by construction — the dir is `root/<id>`, validated to sit
  under the root; reuses the filesystem tool's existing root-confinement.
- ✅ Stays inside the existing tenancy + RBAC; projects are workspace-isolated.
- ✅ Offline-testable: the workspaces root points at a tmp dir in tests; no real
  filesystem or services touched.
- ➖ Local-dir model only for now; a git-backed mode (`repo`) is designed-for but
  not yet implemented.
- ➖ `WORKSPACES_ROOT` must be a writable, ideally persistent volume in
  production (a bind-mount/volume), or projects are lost on container recreate.
