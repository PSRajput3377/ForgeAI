# ADR-0024 — Persist PR approvals in PostgreSQL

- **Status:** Accepted
- **Phase:** 11 (Before-Phase-12 hardening)

## Context
PR approval proposals were initially held in process memory for rapid
iteration. That's fine for a demo but lost on restart, and it can't be shared
across API replicas — neither is acceptable for a durable governance record.

## Decision
Persist PR approvals in a `pr_approvals` PostgreSQL table (async SQLAlchemy,
SQLite in tests — ADR-0018). Each row stores the full `PRPlan` (JSON column) +
repo coordinates + status + `pr_url`, so a proposal survives restarts and can be
approved/executed later by **id alone**. A `PRApprovalStore` wraps the CRUD and
state transitions; the `/github/pr/*` endpoints use it instead of an in-memory
dict.

## Consequences
- ✅ Durable: proposals persist across restarts; auditable governance record.
- ✅ Stateless API: any replica can serve approve/execute (state is in the DB).
- ✅ Clean engineering narrative: "in-memory for iteration → PostgreSQL for
   durability."
- ➖ SQLite returns the status column as a plain string; we coerce to
   `ApprovalStatus` before comparisons (same caveat as ADR-0018).
- ➖ The in-process `ApprovalService` (Phase 8.2) still backs the agent-pipeline
   proposal; the API path is now DB-backed. Unifying them is a future cleanup.
