# ADR-0023 — GitHub writes are human-gated (propose → approve → execute)

- **Status:** Accepted
- **Phase:** 8.2

## Context
An autonomous agent that silently pushes branches and opens PRs is risky and a
weak governance story. Recruiters and enterprises ask "what stops it from
writing the wrong thing?" The answer should be a first-class approval gate, not
a buried `if`.

## Decision
Split the GitHub write path into two phases behind an `ApprovalService`:

1. **`propose(repo, plan)`** — records the intended branch/commit/PR and opens a
   pending `ApprovalRequest`. **Performs no writes.**
2. A human **approves** or **rejects** the request.
3. **`execute(repo, plan, request_id)`** — performs branch → commit → PR **only
   if** the request is approved; otherwise raises `PermissionError` (API → 403).

`GitHubManager` exposes discrete composable ops (`create_branch`,
`create_commit`, `create_pr`, `get_review_status`, `sync_repository`) that the
workflow assembles. The read-only demo surface stays separate and safe.

## Consequences
- ✅ Nothing reaches GitHub without an explicit human decision — provable
  (tests assert no writes occur before approval).
- ✅ Strong security/governance answer; a clean demo (draft → approve → PR).
- ✅ Composable ops enable other gated flows later (merge, deploy).
- ➖ Two API round-trips for a write; intentional — the gate is the point.
- Complements the Phase 6 event-driven HumanApprovalCenter (UI streaming);
  request ids can link the two.
