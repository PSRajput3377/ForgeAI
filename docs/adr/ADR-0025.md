# ADR-0025 — Evaluation substrate: versioned rubric over an append-only record

- **Status:** Accepted
- **Phase:** 12.1 (Self-Improving Agent System — Evaluation Engine)

## Context
Phase 12 turns the static pipeline into a system that improves across runs. Every
downstream component — per-agent stats, benchmarks, prompt comparison, the
deferred learning loops — needs a common, comparable unit of measurement. Today a
run ends with a pass/fail verdict and nothing durable that says *how good* it was
or *what it cost*. The learning loops themselves depend on real usage data we do
not have yet, so the foundation must stand on its own and be fully testable
offline (see `specs/self-improvement-spec.md`).

## Decision
Introduce a `packages/evaluation` package that, after each run, derives one
`Evaluation` record (outcome + cost + provenance) from the final `ProjectState`
via a **versioned rubric**:

- `score` is a documented function in [0, 1]; its `rubric_version` is stamped on
  every record. Changing weights means a **new** named rubric (`v1` → `v2`),
  never an edit — so historical scores stay interpretable.
- Records land in an `EvaluationStore` — in-memory (12.1) and a PostgreSQL-backed
  `PerformanceStore` (12.2, the `evaluations` table) behind the same shape,
  mirroring how `observability/store.py` evolved (ADR-0017). Both compute their
  aggregates from the records via the same pure `evaluation.stats` functions, so
  the two back-ends agree by construction.
- Per-agent aggregates are **derived** from the records (mean score, success
  rate, prompt-version slices), never a separate writable table that can drift
  (spec §2).
- The workflow accepts an optional `evaluation_store`; when absent, behavior is
  identical to before. This matches the existing `bus` / `github_workflow`
  opt-in seams — the offline default path is unchanged.

## Consequences
- ✅ One comparable unit underpins all of Phase 12; nothing downstream invents
  its own scoring.
- ✅ Fully deterministic and offline-testable (`EchoModel`, in-memory store) — no
  models or services required.
- ✅ Versioned rubric keeps "we changed how we score" from silently corrupting
  trend lines.
- ➖ `v1` folds only success, review score, and retries into `score`; efficiency
  (time/tokens) is recorded but not yet weighted — deferred to a `v2` rubric.
- ➖ The review score is parsed best-effort from free-text review feedback; a
  structured review output (a later sub-phase) would make it exact.
- ➖ `pr_accepted` is carried on the record but always null until the GitHub
  backfill seam is wired (spec §10, sub-phase 12.9).
